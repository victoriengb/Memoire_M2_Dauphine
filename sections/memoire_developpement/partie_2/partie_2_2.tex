\subsection{\textit{Continuous Benchmarking} pour le \textit{matching engine}}

\subsubsection{Contexte général}

Pour le développement de notre \textit{matching engine}, nous avons adopté le principe du \textit{continuous benchmarking}, qui se distingue de l'intégration continue classique en intégrant une phase de validation des performances à chaque jalon, garantissant qu'aucune modification du code ne dégrade silencieusement les acquis des itérations précédentes. Cette exigence empêche, dans un contexte de trading haute fréquence, qu'une régression de quelques microsecondes passe inaperçue. Cela représenterait un risque financier quantifiable, et non un simple inconfort de performance. La référence sur laquelle s'appuient les comparaisons est définie manuellement, afin d'éviter qu'un commit non pertinent, un refactoring mineur ou une correction de documentation, ne vienne silencieusement déplacer cette référence et propager du bruit de mesure vers les comparaisons futures.

Concernant la métrique retenue, nous ciblons la latence P99.99. L'idéal aurait été d'utiliser le percentile P99.995, cohérent avec les exigences non-fonctionnelles définies en section \ref{exigences-nf} ; ce percentile n'étant pas disponible nativement au sein de JMH, nous l'intégrerons ultérieurement via les histogrammes HDR.

Enfin, la politique de régression repose sur deux seuils : au-delà de 20\%, le code est rejeté, signalant une rupture structurelle incompatible avec nos objectifs ; entre 0\% et 20\%, un avertissement est émis. Ce palier assure une flexibilité opérationnelle en signalant les dérives sans bloquer les contributions dont le bénéfice global reste positif.

\subsubsection{Architecture du pipeline de \textit{Continuous Benchmarking}}

Le pipeline a pour objectif de garantir qu'aucune régression de performance ne franchit la barrière de la revue de code. Il s'exécute sur GitHub Actions à partir de trois déclencheurs : un \textit{push} sur \texttt{main}, une \textit{pull request}, ou un déclenchement manuel (\textit{workflow\_dispatch}). Ce dernier mode est essentiel : il permet de rejouer le pipeline sur un commit précis, indépendamment de l'historique, afin de valider ou d'invalider manuellement une baseline candidate.

À l'échelle macro, le pipeline est structuré en trois jobs séquentiels à responsabilités disjointes, comme l'illustre la figure~\ref{fig:pipeline-continuous-benchmarking}. À l'échelle micro, chaque job délègue à un outil spécialisé :

\begin{itemize}
    \item \textbf{Job 1 --- \textit{Correctness gate}} (\texttt{build-and-test}) : Maven compile le projet et exécute la suite de tests unitaires via \texttt{mvn -B verify}. Les jobs de performance ne s'exécutent que si ce verrou de correction fonctionnelle est franchi.
    \item \textbf{Job 2 --- \textit{Performance gate}} (\texttt{benchmark}) : les tests unitaires sont délibérément sautés (\texttt{-DskipTests}) pour ne pas dupliquer le travail du job précédent. JMH exécute les benchmarks et produit un fichier \texttt{jmh-result.json} au format JSON natif, publié en artefact de pipeline afin de découpler la production du résultat de son analyse.
    \item \textbf{Job 3 --- \textit{Regression gate}} (\texttt{compare-baseline}) : le script \texttt{compare-benchmarks.sh} télécharge l'artefact et le compare à la baseline versionnée dans le dépôt (\texttt{benchmarks/baseline.json}). Le résultat est publié dans le \textbf{Job Summary} de GitHub Actions, directement visible dans l'interface du run sans permissions supplémentaires.
\end{itemize}

\subsubsection{Choix d'implémentation du pipeline \textit{Continuous Benchmarking}}

Le script \texttt{compare-benchmarks.sh} implémente une barrière à l'intégration du code (logique de \textit{gate}) en s'appuyant sur deux utilitaires natifs du \textit{runner} Linux, sans dépendance externe. À l'échelle macro, il orchestre trois phases : extraction des percentiles, calcul du taux de régression, puis application de la politique de seuils. À l'échelle micro, chaque phase délègue à un outil adapté.

\textbf{Extraction des percentiles avec \texttt{jq}.} Le format de sortie natif de JMH (\texttt{-rf json}) expose les latences par percentile sous la clé \texttt{primaryMetric.scorePercentiles}, indexée par valeur numérique (ex. \texttt{``99.99''}). L'extraction repose sur une approche déclarative plutôt qu'impérative : on exprime directement \textit{quoi} chercher, le percentile clé d'un benchmark nommé, sans écrire la boucle ou les indices qui parcourraient explicitement la structure JSON. \texttt{jq} se charge seul de cette traversée et restitue directement la valeur en \texttt{ns/op}, sans variable intermédiaire de type tableau.

\begin{lstlisting}[language=bash, caption={Extraction déclarative d'un percentile via jq}]
jq -r --arg bench "$benchmark_name" --arg pct "$percentile_key" '
    .[] | select(.benchmark == $bench) | .primaryMetric.scorePercentiles[$pct]
' "$file"
\end{lstlisting}

\textbf{Calcul du taux de régression avec \texttt{awk}.} Bash ne disposant pas d'arithmétique flottante native, la variation $((curr - base) / base) \times 100$ est déléguée à \texttt{awk}. Une garde sur \texttt{base == 0} évite la division par zéro lors d'un premier run à latence nulle.

\begin{lstlisting}[language=bash, caption={Calcul du taux de régression via awk}]
    awk -v base="$baseline_value" -v curr="$current_value" '
        BEGIN {
            if (base == 0) { print "0"; exit }
            printf "%.2f", ((curr - base) / base) * 100
        }
    '
\end{lstlisting}

\textbf{Application de la politique de seuils.} Deux indicateurs booléens pilotent l'issue du job : \texttt{is\_fail}, positif si la régression dépasse le seuil d'échec (20\%), et \texttt{is\_regression}, positif dès qu'une dérive positive est détectée. Cette séparation permet de distinguer les régressions bloquantes des dérives à surveiller sans interrompre le développement.

\begin{lstlisting}[language=bash, caption={Flux principal de non-régression au niveau des benchmarks}]
    local is_fail
    is_fail=$(awk -v r="$regression_pct" -v t="$REGRESSION_FAIL_THRESHOLD_PCT" 'BEGIN { print (r > t) ? "1" : "0" }')

    local is_regression
    is_regression=$(awk -v r="$regression_pct" 'BEGIN { print (r > 0) ? "1" : "0" }')

    if [[ "$is_fail" == "1" ]]; then
        write_summary "**Échec : régression de ${regression_pct}% > seuil de ${REGRESSION_FAIL_THRESHOLD_PCT}%.**"
        echo "ÉCHEC : régression de ${regression_pct}% (seuil : ${REGRESSION_FAIL_THRESHOLD_PCT}%)." >&2
        exit 1
    elif [[ "$is_regression" == "1" ]]; then
        write_summary "**Avertissement : régression de ${regression_pct}%, sous le seuil d'échec mais à surveiller.**"
        echo "AVERTISSEMENT : régression de ${regression_pct}%, sous le seuil d'échec."
        exit 0
    else
        write_summary "Pas de régression détectée (variation : ${regression_pct}%)."
        echo "OK : variation de ${regression_pct}% (amélioration ou stabilité)."
        exit 0
    fi
\end{lstlisting}

Un cas particulier est géré explicitement : l'\textbf{absence de baseline} (\textit{bootstrap}). Lors du premier run d'un nouveau jalon, aucun fichier de référence n'existe encore dans le dépôt. Le script détecte cette situation et publie le résultat courant comme candidat à la promotion, sans déclencher d'échec. Promouvoir ce run en nouvelle baseline se résume à versionner le fichier résultat : \texttt{cp jmh-result.json benchmarks/baseline.json}. Ce mécanisme garantit que la baseline est toujours un artefact explicitement validé par le développeur, et non un résultat produit automatiquement.

Cette discipline de garde explicite, intercepter une absence avant d'agir, plutôt que de laisser une erreur silencieuse se propager, n'est pas isolée au cas de la baseline : elle se retrouve à l'identique sur l'absence du fichier de résultat courant en tout début de script, et sur la division par zéro dans le calcul du taux de régression. C'est moins une décision ponctuelle qu'un idiome récurrent de programmation défensive, appliqué uniformément à chaque point d'entrée incertain du script.

\textbf{Alternatives écartées.} Trois choix d'implémentation ont été écartés, chacun au prix d'une renonciation explicite. L'action communautaire github \texttt{benchmark-action/github-action-benchmark} effectue une comparaison automatique avec le \textit{run} précédent et propose nativement un tableau de bord historique ainsi que des commentaires automatiques sur les \textit{pull requests} ; elle a néanmoins été écartée car ce modèle de comparaison est incompatible avec notre logique de baseline manuelle, où seuls les jalons explicitement validés servent de référence --- la renonciation se paie donc en visualisation automatique de l'historique, que nous reconstituerons manuellement si besoin pour la soutenance. Python a également été écarté au profit des utilitaires natifs du \textit{runner} Linux (\texttt{jq}, \texttt{awk}), pour conserver un script auto-contenu sans gestion de dépendances ; ce choix se paie en lisibilité, l'absence d'arithmétique flottante native en bash forçant à fragmenter le calcul de régression entre deux outils plutôt que de le centraliser dans un seul langage. Enfin, la publication du résultat a été orientée vers le \textbf{Job Summary} plutôt que vers un commentaire automatique sur la \textit{pull request} : cette dernière option aurait offert une visibilité plus immédiate dans le flux de revue, mais elle nécessite des permissions d'écriture étendues sur le \textit{token} GitHub, qui échouent silencieusement sur les \textit{pull requests} issues d'un \textit{fork}, un risque jugé plus coûteux que le léger surcroît de clics pour consulter le Job Summary.
